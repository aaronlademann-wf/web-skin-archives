Download Web Skin

Web Skin has a few easy ways to quickly get started, each one appealing to a different skill level and use case. Read through to see what suits your particular needs.

Compiled CSS, JS, and fonts

The fastest way to get Web Skin is to download the compiled and minified versions of our CSS and JavaScript, along with the included fonts. No documentation or original source files are included.

Download Web Skin

Additional downloads

Download source code

Get the original files for all CSS and JavaScript by downloading the full release directly from GitHub.

Clone or fork via GitHub

Clone the entire project or fork your own version of web-skin to make it your own by visiting us on GitHub.

Install with Bower

Install and manage Web Skin via the web-skin platform using Bower.

First, install Bower.

$ sudo npm install bower -g

To define web-skin as a dependency for your application, Add a bower.json file to the root of your project like so:

{
    "name": "Your App Name",
    "version": "your.app.version",
    "dependencies": {
        "web-skin": "git+ssh://git@github.com/Workiva/web-skin.git#0.4.11"
    }
}

Then as part of your build setup / grunt targets, run the following command in bash:

$ bower install

All of the package resources will be pulled into the root of your project/app in a folder named bower_components/web-skin/

What’s included

Within the download you’ll find the following directories and files, logically grouping common assets and providing both compiled and minified variations.

Once downloaded, unzip the compressed folder to see the structure of Web Skin. You'll see something like this:

dist/
│
├── css/
├── fonts/
├── ico/
├── img/
├── js/
├── cachebuster.txt
├── shasum.txt
└── version.txt

This is the most basic form of Web Skin: compiled files for quick drop-in usage in nearly any web project. We provide compiled CSS and JS.

Core Lib JS Required

Please note that Web Skin was designed under the assumption that its consumer applications would be including the JavaScript assets in wf-vendor.js and libs.js/libs_ie.js in their project(s). See the basic template section below for how these should be included in your HTML.

Extending with Sass

Make building your app’s Sass easier by extending Web Skin’s built-in Sass variables, mixins, functions and more.

Set up your app to compile Sassy CSS

Need help extending Web Skin Sass into your app?

If you need help or some clarification on how to proceed with the steps provided below, please give us a shout on HipChat or open an issue.

1. Install Web Skin with Bower

After you install Web Skin using Bower, you’re ready to set up your app’s Sass using the steps provided below.

2. Add the Ruby Gems you’ll need

There are many ways to install Ruby Gems. However, we highly recommend addign a Gemfile to your app so that you can declare/bundle your app’s dependencies - and use the bundle exec option when running your build in a CI environment.

Create your Gemfile

source 'https://rubygems.org'

gem 'json'
gem 'compass', '0.12.2'
gem 'sass', '3.2.12'

Bundle your dependencies

Local Dev Machine:

bundle

Smithy CI:

bundle config --local PATH ./.rubygems
export PATH=$PATH:./.rubygems/bin
bundle

3. Set up your Compass config

Now that you have Sass and Compass installed, the next thing you need to do is make sure your Compass configuration file knows where to find the Web Skin Sass files. Add the following line to your config.rb file.

additional_import_paths = ["bower_components/web-skin/sass"]

4. Set up your Gulp/Grunt Compass task

Next, you need to require a file containing some custom Ruby functions that help Web Skin scale.

Using Grunt

We recommend using the grunt-contrib-compass plugin to handle Sass compilation in a Grunt build setup. Below is an example showing the recommended Grunt config options for your app.

grunt.initConfig({
    compass: {
        app: {
            options: {
                /*
                    highly recommend using `bundleExec` to keep control
                    of Ruby Gems in the hands of your team...
                    not the hands of CI server maintainers.
                */
                bundleExec: true,
                config: './config.rb',
                require: './bower_components/web-skin/lib/ruby/sass-functions.rb'
            }
        }
    }
});

grunt.loadNpmTasks('grunt-contrib-compass');

Using Gulp

We recommend using the gulp-compass plugin to handle Sass compilation in a Gulp build setup. Below is an example showing the recommended Gulp config options for your app.

var compass = require('gulp-compass');

gulp.task('compass', function() {
    gulp.src('./' + sassSourceFileDir + '/*.scss');
    .pipe(compass({
        /*
            highly recommend using `bundle_exec` to keep control
            of Ruby Gems in the hands of your team...
            not the hands of CI server maintainers.
        */
        bundle_exec: true,
        config_file: './config.rb',
        require: './bower_components/web-skin/lib/ruby/sass-functions.rb',
        css: cssOutputFileDir,
        sass: sassSourceFileDir
    }));
});

5. Import Web Skin Sass

Once you’ve added the import paths to your Compass config, and required the custom Sass functions required for Web Skin, you can import the Web Skin Sass constants in your Sass file so they can be referenced throughout your app’s Sass files, and your app will update any time the values change in Web Skin.

//
// Any default Web Skin values you want
// to override should be placed BEFORE
// the import of the Web Skin manifest
//
// For example, if you do not want your stylesheet
// to have MSIE specific hacks/support...
//
$ie-support: false;

//
// Web Skin Constants
//
@import "_constants/_wdesk-manifest.scss";

// Now you can write some responsive styles using the same settings
// as the global packages uses so you're always in lock-step!
@media (min-width: $screenTablet) and (max-width: $screenTabletMax) {
    // mobile styles here
}

// Or make an element in your app be "Wdesk" Green
.my-element {
    background-color: $wf_green;
}

Additional notes

We’ve made many underlying changes in 0.2.x that are not immediately apparent. Base classes, key styles, and behaviors have been adjusted for flexibility and our mobile first approach.

Navs

  • You can no longer add "disabled" appearance to a nav link/button by using the parent <li class="disabled"> element. Instead, add the disabled class to an <a> element, or add the disabled attribute to a <button> element.

Forms

  • Text-based form controls no longer receive styling without adding the .form-control class.
  • Text-based form controls are now 100% wide, and cannot have grid classes applied directly to them. Wrap inputs inside <div class="col-*"></div> to control input widths.
  • .form-horizontal control groups no longer receive default widths. Use .form-group elems as grid rows, and wrap your inputs in grid spans to enforce a specific width as shown in this form-horizontal example.

Modals

  • Modal markup has changed significantly. The .modal-header, .modal-body, and .modal-footer sections now get wrapped in .modal-content and .modal-dialog for improved mobile styling and behavior.

Accordions

  • The accordion element classes shown in the Collapse Plugin JavaScript Docs of v0.1.x are no longer in-use. Instead, use the new Panels component to build accordions.

Javascript

  • JavaScript events are namespaced. For example, to handle the modal "show" event, use 'show.wdesk.modal'. For tabs "shown" use 'shown.wdesk.tab', etc.

Additional notes

Global statusbar

  • The .statusbar-left, .statusbar-middle and .statusbar-right elements within the global statusbar component should use .col-xs-* grid columns so that they never "collapse" no matter the screen size. See this commit or Issue #358 to reference the changes you’ll need to make in your app’s component.

Paring down CSS selectors

To continue providing a single CSS file for all Web Skin components, some CSS selector bloat had to be removed (thanks to MSIE’s wonderful 4095 selector limit). In order to accomplish this, the following changes have been made:

  • Alert elements are no longer blue by default. Use the .alert-alt CSS class to colorize alerts in that manner.
  • Using <fieldset disabled> is no longer supported. Disable individual form controls within a fieldset to disable them.
  • Using <td> elements within <thead> elements is no longer supported. Using <th> is more semantically correct, anyway.
  • Selectable table rows with ng-click or wf-click attributes will no longer inherit the "pointer" mouse cursor appearance. Use the .clickable CSS class on those elems instead.
  • Using responsive visibility classes on inline elements now requires the use of the .inline CSS class.
  • Form controls and progress bars no longer have *-default or *-primary state classes. To get the green color of *-primary, use *-success instead.

Web Skin does not support any styling your customer (or your team’s dev QA) sees when viewing a Wdesk app in Internet Explorer compatibility modes. To be sure you’re using the latest rendering mode for IE, consider including the appropriate <meta> tag in your app/page:

<!-- MUST BE THE FIRST META TAG WITHIN <HEAD> -->
<meta http-equiv="X-UA-Compatible" content="IE=edge">

Windows Phone 8 and MSIE 10+

Internet Explorer 10+ doesn’t differentiate device-width from viewport-width, and thus doesn’t properly apply the media queries in Web Skin’s CSS. To address this, you can add the following JavaScript to work alongside the CSS that is included with Web Skin until Microsoft issues a fix (don’t hold your breath).

/**
 *  Add this to work alongside Web Skin&rsquo;s CSS
 */
if ("-ms-user-select" in document.documentElement.style &&
 navigator.userAgent.match(/IEMobile/ ||
 navigator.userAgent.match(/ZuneWP7/) ||
 navigator.userAgent.match(/WPDesktop/))) {
    var msViewportStyle = document.createElement("style");
    msViewportStyle.appendChild(
        document.createTextNode(
            "@-ms-viewport { width: auto !important; }"
        )
    );
    document.getElementsByTagName("head")[0].
        appendChild(msViewportStyle);
}
/**
 *  This comes with wdesk_ie.css
 */
@-ms-viewport {
    width: device-width;
}

For more information and usage guidelines, read Windows Phone 8 and Device-Width.

Browser zooming

Page zooming inevitably presents rendering artifacts in some components, both in Web Skin and the rest of the web. Depending on the issue, we may be able to fix it (search first and then open an issue if need be).

However, we tend to ignore these as they often have no direct solution other than hacky workarounds. The zoom-related suggestions we can make are listed below.

Disable touch device zoom

In order to completely disable zoom capabilities on a touch device, place the following code in the <head> of your document:

<meta name="viewport" content="width=device-width,
                               initial-scale=1.0,
                               maximum-scale=1.0,
                               target-densityDpi=device-dpi">

MSIE 11 touch device zoom

All other browsers conveniently control the zoom capabilities of a page via the <meta name="viewport"> method described above. However, MSIE 11 is special (and always has been… “special”).

Disabling zoom via a user’s “double-tap” gesture:
body {
    -ms-touch-action: manipulation !important;
}
Disabling all zoom capabilities:
body {
    -ms-content-zooming: none !important;
}

Modals and mobile devices

Overflow and scrolling

Support for overflow: hidden on the <body> element is quite limited in iOS and Android. To that end, when you scroll past the top or bottom of a modal in either of those devices’ browsers, the <body> content will begin to scroll.

Virtual keyboards

Also, note that if you’re using inputs in your modal – iOS has a rendering bug that doesn’t update the position of fixed elements when the virtual keyboard is triggered. A few work arounds for this include transforming your elements to position: absolute or invoking a timer on focus to try to correct the positioning manually. This is not handled by Web Skin, so it is up to you to decide which solution is best for your application.

Related: Fixed Navbar & iOS Virtual Keyboard

Disabling responsiveness

Don’t want your application to be scaled on different devices? With a little bit of work, you can disable the responsive features of Web Skin so that mobile users see your full desktop-version site.

Steps to disable responsive views

  1. Remove (or just don’t add) the viewport <meta> mentioned in the CSS docs
  2. Remove the max-width on the .container for all grid tiers with max-width: none !important; and set a regular width. Be sure that this comes after the default Web Skin CSS. You can optionally avoid the !important with media queries or some selector-fu.
  3. If using navbars, undo all the navbar collapsing and expanding behavior.
  4. For grid layouts, make use of .col-xs-* classes in addition to or in place of the medium/large ones. Don’t worry, the extra-small device grid scales up to all resolutions, so you’re set there.

You'll still need respond.js for MSIE8 (since our media queries are still there and need to be picked up).

// Reset the container
.container {
    max-width: none !important;
    width: $screenDesktop;
}

.container .navbar-header,
.container .navbar-collapse {
    margin-right: 0;
    margin-left: 0;
}

// Always float the navbar header
.navbar-header { float: left; }

// Undo the collapsing navbar
.navbar-collapse {
    display: block !important;
    height: auto !important;
    padding-bottom: 0;
    overflow: visible !important;
}
.navbar-toggle { display: none; }
.navbar-brand { margin-left: -#{$gridGutterWidth / 2}; }

// Always apply the floated nav
.navbar .nav {
    float: left;
    margin: 0;
    > li {
        float: left;
        > .hitarea {
            padding: 0 nth($navbarPadding, 2);
        }
    }
}

// Redeclare since we override the float above
.navbar .nav.navbar-right { float: right; }

Accessibility

Web Skin follows standards set by both the A11y Project and WCAG 2.0 . With minimal extra effort, Web Skin can be used to create Wdesk apps that are accessibile to those using AT.

Why Should You Care About Accessibility?

Excerpts from “Tackling Accessibility On The Web”

There are three main reasons why Workiva developers should care about making our sites and apps accessible.

#1  It’s the right thing to do

This reason alone should be all you need to convince you that accessibility is worth the time and effort.

Building accessible websites and applications is just simply the right thing to do.

As developers we have enormous power to create anything that our minds can dream of. It’s our responsibility to make sure that the things that we create can be enjoyed by everyone, regardless of whether they might have a disability.

#2  We are gaining clients that are bound to accessibility by law

As Workiva continues to grow and expand into the private sector, some clients (mostly Governments) will require that our applications comply with Section 508 of the Rehabilitation Act. It requires that all Federal agencies make digital information accessible to everyone.

#3  You will become better at what you do

Whether you’re a software engineer, designer, copywriter or any other Workiva team member working on improving customer experiences, having a focus on accessibility will help you to improve your skills.

Lets look at the example of a software engineer. Building accessible web applications means that you need to write good, clean, standards-compliant code. That point in itself has a lot of benefits outside of the realm of accessibility. Writing good code that complies with W3C and WHATWG standards means that it is going to be easier for others to understand, which is a big bonus since we are all part of a larger team of Workiva engineers.

Understanding the extremes

Industrial designer Dan Formosa once said:

What we really need to do to design is look at the extremes.

The weakest, or the person with arthritis, or the athlete, or the strongest person.

Because if we understand what the extremes are, the middle will take care of itself.

That last line is particularly powerful. When we are designing we need to think about the power users. How can we make using our application easier for people that spend significant amounts of time using it? Keyboard shortcuts are a good example of designing for power users.

We also need to think about people at the other extreme end of the spectrum. What can we do to make our apps easier to navigate for blind people that rely on screen readers? Will somebody that has trouble moving a mouse precisely be able to click that small button?

Asking all these questions forces you to think more deeply about whatever it is that you are building. It develops your understanding of problems and expands your skills as a web professional.

Helper Classes & Best Practices

In addition to the Web Skin components listed above, here are some examples of some global CSS classes available in Web Skin, along with some best-practices being put to good use.

Add context for screen readers

Layout and proximity are methods often used by UI architects to ensure that more succinct wording, or even a lack of wording can be used without losing context. However, screen readers do not have the ability to communicate that contex to a user unless we as developers explicitly provide it.

Using a combination of the .sr-only CSS helper class and the aria-hidden attribute can help us do just that.

Take an Alert dialog “close” button for example:

  • Anyone who can see the  ×  iconography generally undestands what will happen when clicked.
  • Those using assistive technologies like screen readers - will not understand this, and a critical piece of UI functionality will be inaccessible to them unless we explicitly tell the user what this button does using the markup shown below.

Alert text…

<!--
    Screen Readers will ignore anything with `aria-hidden="true"`
    Screen Readers will read the text within `.sr-only` to the user
-->
<button type="button" class="close" data-dismiss="alert">
    <i aria-hidden="true">&times;</i>
    <i class="sr-only">Close</i>
</button>

Skip navigation

Another way to utilize the .sr-only CSS helper class is to hide a link that enables screen readers to skip past navigation structures containing many links when that structure comes before the main content in the DOM.

Add a Skip to content link immediately after your opening <body> tag. (read why, or read the whole WCAG 2.0 spec)

<body>
    <a href="#content" class="sr-only sr-only-focusable">Skip to content</a>
    <div class="container" id="content">
        The main page content.
    </div>
</body>

Tabindex

If using a DOM element other than an <input>, <button>, <textarea>, <select> or <a> - you must add tabindex="0" to it if you want the user to be able to navigate to it using their keyboard.

This is an especially important concept when making keyboard navigation possible in our JS plugins.

<li class="dropdown open">
    <ul role="menu" class="dropdown-menu">
        <li role="presentation">
            <a href="#content">Anchor link</a>
        </li>
        <li role="presentation">
            <button type="button">Button action</button>
        </li>
        <li role="presentation">
            <!--
                Without this tabindex, this menu structure would
                not be able to be traversed using a keyboard.
            -->
            <div role="button" tabindex="0">Div posing as a button</div>
        </li>
    </ul>
</li>

Nested headings

When nesting headings (<h1> - <h6>), your primary document header should be an <h1>. Subsequent headings should make logical use of <h2> - <h6> such that screen readers can construct a table of contents for your pages.

Read more about nesting headings: Penn State’s AccessAbility
