Overview

Get the lowdown on the foundation of the Web Skin platform, including our approach to better, faster, stronger web application development.

HTML5 doctype required

Web Skin makes use of certain HTML elements and CSS properties that require the use of the HTML5 doctype. Include it at the beginning of all your Wdesk projects.

<!DOCTYPE html>
<html lang="en">
    ...
</html>

Mobile first

Web Skin is written to be mobile friendly from the start. Instead of adding on optional mobile styles, they’re baked right into the core. In fact, Web Skin is mobile first. Mobile first styles can be found throughout the entire library instead of in separate files.

To ensure proper rendering and touch zooming, add the viewport meta tag to your <head>.

<meta name="viewport" content="width=device-width, initial-scale=1.0">

Touch Friendly

Web Skin’s patterns have been designed and implemented to make the user experience just as enjoyable on a touch interface as it is on a traditional mouse/keyboard interface.

To ensure proper rendering of all interactions on a touch device, add the ontouchstart attribute to your <body>.

<body ontouchstart>
    ...
</body>

Responsive images

Images in your Wdesk app can be made responsive-friendly via the addition of Web Skin’s .img-responsive class. This applies max-width: 100%; and height: auto; to the image so that it scales nicely to the parent element.

<img src="..." class="img-responsive" alt="Reponsive image">

Typography and links

Web Skin sets basic global display, typography, and link styles. Specifically, we:

  • Remove margin on the body
  • Set background-color: $bodyBg; on the body
  • Use the $fontFamilyBase, $fontSizeBase, and $lineHeightBase Sass variables as our typographic base
  • Set the global link color via $linkColor and apply link underlines and $linkColorHover only on :hover

These styles can be found within _scaffolding.scss.

Normalize reset

For improved cross-browser rendering, we use Normalize, a project by Nicolas Gallagher and Jonathan Neal.

Grid system

Web Skin includes a responsive, mobile-first fluid grid system that appropriately scales up to columns as the device or viewport size increases. It includes predefined classes for easy layout options, as well as powerful mixins for generating more semantic layouts.

Introduction

Grid systems are used for creating page layouts through a series of rows and columns that house your content. Here’s how the Web Skin grid system works:

  • Rows must be placed within a .container (fixed-width) or .container-wide (full-width) for proper alignment and padding.
  • Use .rows to create horizontal groups of columns.
  • Content should be placed within columns, and only columns may be immediate children of rows.
  • Predefined grid classes like .row and .col-xs-4 are available for quickly making grid layouts. Sass mixins can also be used for more semantic layouts.
  • Columns create gutters (gaps between column content) via padding. That padding is offset in rows for the first and last column via negative margin on .rows.
  • Grid columns are created by specifying the number of available columns you wish to span. For example, three equal columns would use three .col-xs-4 elements.

Look to the grid examples for applying these principles to your code.

Media queries

We use the following media queries to create the key breakpoints in our grid system.

// Extra small devices (phones, up to 480px)
// No media query since this is the default in Web Skin (mobile first)

// Small devices (tablets, 768px and up)
@media (min-width: $screenTablet) { ... }

// Medium devices (tablets, 992px and up)
@media (min-width: $screenDesktop) { ... }

// Large devices (large desktops, 1200px and up)
@media (min-width: $screenDesktopLarge) { ... }

// Huge devices (huge desktops, 1600px and up)
@media (min-width: $screenDesktopHuge) { ... }

We occasionally expand on these media queries to include a max-width to limit CSS to a narrower set of devices.

@media (max-width: $screenPhoneMax) { ... }
@media (min-width: $screenTablet) and (max-width: $screenTabletMax) { ... }
@media (min-width: $screenDesktop) and (max-width: $screenDesktopMax) { ... }
@media (min-width: $screenDesktopLarge) and (max-width: $screenDesktopLargeMax) { ... }
@media (min-width: $screenDesktopHuge) { ... }

    <div class="col-xs-6 col-sm-3">.col-xs-6 .col-sm-3</div>

    <!-- Add the extra clearfix for only the required viewport -->
    <div class="clearfix visible-xs"></div>

Sass mixins and variables

In addition to prebuilt grid classes for fast layouts, Web Skin includes Sass variables and mixins for quickly generating your own simple, semantic layouts.

Variables

Variables determine the number of columns, the gutter width, and the media query point at which to begin floating columns. We use these to generate the predefined grid classes documented above, as well as for the custom mixins listed below.

$gridColumns:             12;
$gridGutterWidth:         30px;
$gridFloatBreakpoint:     768px;

Mixins

Mixins are used in conjunction with the grid variables to generate semantic CSS for individual grid columns.

// Creates a wrapper for a series of columns
@mixin makeRow($gutter: $gridGutterWidth) {
    // Then clear the floated columns
    @include clearfix();

    @media (min-width: $screenTablet) {
        margin-left:  ($gutter / -2);
        margin-right: ($gutter / -2);
    }

    // Negative margin nested rows out to align the content of columns
    .row {
        margin-left:  ($gutter / -2);
        margin-right: ($gutter / -2);
    }
}

// Generate the extra small columns
@mixin makeXsColumn($columns, $gutter: $gridGutterWidth) {
    position: relative;
    // Prevent columns from collapsing when empty
    min-height: 1px;
    // Inner gutter via padding
    padding-left:  ($gutter / 2);
    padding-right: ($gutter / 2);

    // Calculate width based on number of columns available
    @media (min-width: $gridFloatBreakpoint) {
        float: left;
        width: percentage(($columns / $gridColumns));
    }
}

// Generate the small columns

Typography

Everything from our standard headings to body copy, blockquotes, lists and more!

Typographic scale

Powered by Sass: The typographic scale is based on two Sass variables in _variables.scss: $fontSizeBase and $lineHeightBase. The first is the base font-size used throughout and the second is the base line-height.

We use those variables and some simple math to create the margins, paddings, and line-heights of all our typography, components and more. Customize them and Web Skin adapts.

Gibson font usage

The Gibson font family is used in Wdesk apps to reinforce the Workiva brand. By nature, it is a titling font, looking it’s best at larger font sizes. To keep it looking it’s best, and to ensure that it has maximum “impact” when it appears in your page / application, it should be used only when combined with a font-size of 16px or higher.

The most basic example of this is within headings <h1>, <h2>, <h3> and <h4> as demonstrated in the headings documentation. If you are unsure whether or not you should be using Gibson for a particular design / layout you’re working on, contact us on HipChat and we’ll try to help clarify.

Gibson usage in MSIE 8 or below

Because Gibson cannot be properly aliased by Internet Explorer 8 or below, all headings will use the native Helvetica/Arial sans-serif font.

See Issue #607 for more information.

Supported table markup

Below is a list of supported table HTML elements and how they should be used.

This table lists the HTML tags that are supported within table elements when using the Web Skin platform.
Tag Description
<table> Wrapping element for displaying data in a tabular format
<thead> Container element for table header rows
<tbody> Container element for table body rows
<tr> Container element for a row of one or more table cells
<td> Default table cell (nested within <tbody> elem only)
<th>

Heading table cell used to label columns or rows, depending on scope and placement. Use of the scope attribute is especially important for accessibility.

<table class="table">
    <!-- th cells in thead rows should be scoped for columns -->
    <thead>
        <tr>
            <th scope="col">Name</th>
            <th scope="col">Location</th>
            <th scope="col">Job Title</th>
        </tr>
    </thead>
    <!-- th cells in tbody rows should be scoped for rows -->
    <tbody>
        <tr>
            <th scope="row">Jason Moore</th>
            <td>Scottsdale, Arizona</td>
            <td>UX Lead</td>
        </tr>
    </tbody>
</table>
<caption> Recommended Description or summary of what the table holds, especially useful for screen reader accessibility when combined with our .sr-only CSS helper class
<colgroup> Optional Specifies a group of one or more columns in a table for formatting
<col> Optional Specifies column properties for each column within a <colgroup> element

Scroll utilities

Use these utility classes to control the overflow of content in your components.

.scroll-x

If the contents of an element with this class exceeds the width of the element, a horizontal scrollbar appears.

.scroll-x-force

An element with this class always displays a horizontal scrollbar.

.scroll-y

If the contents of an element with this class exceeds the height of the element, a vertical scrollbar appears.

.scroll-y-force

An element with this class always displays a vertical scrollbar.

.scroll

If the contents of an element with this class exceeds the width or height of the element, a scrollbar appears in the direction of the aforementioned overflow.

Make carets accessible

Since carets don’t have any semantic meaning without being able to visually observe the triangular shape of the element, be sure to add a aria-hidden="true" to every .caret element to ensure WCAG 2.0 compliance.

Screen reader and keyboard navigation content

Hide an element to all users except screen readers with .sr-only.

Necessary for following accessibility best practices. Can also be used as a Sass mixin.

<button type="button" class="close">
    <i aria-hidden="true">&times;</i>
    <i class="sr-only">Close</i>
</button>
// Usage as a Sass Mixin
.your-custom-class {
  @include sr-only;
}

Combine .sr-only with .sr-only-focusable to show the element again when it’s focused (e.g. by a keyboard-only user). Can also be used as a Sass mixin.

<a class="sr-only sr-only-focusable" href="#content">Skip to main content</a>
// Usage as a Sass Mixin
.skip-navigation {
  @include sr-only;
  @include sr-only-focusable;
}
