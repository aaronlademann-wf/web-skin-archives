Download Web Skin

Web Skin has a few easy ways to quickly get started, each one appealing to a different skill level and use case. Read through to see what suits your particular needs.

Compiled CSS, JS, and fonts

The fastest way to get Web Skin is to download the compiled and minified versions of our CSS and JavaScript, along with the included fonts. No documentation or original source files are included.

Clone or fork via GitHub

Clone the entire project or fork your own version of web-skin to make it your own by visiting us on GitHub.

Install with Bower

Install and manage Web Skin via the web-skin platform using Bower.

First, install Bower.

$ sudo npm install bower -g

To define web-skin as a dependency for your application, Add a bower.json file to the root of your project like so:

{
    "name": "Your App Name",
    "version": "your.app.version",
    "dependencies": {
        "web-skin": "git+ssh://git@github.com/WebFilings/web-skin.git#0.2.34"
    }
}

Then as part of your build setup / grunt targets, run the following command in bash:

$ bower install

All of the package resources will be pulled into the root of your project/app in a folder named bower_components/web-skin/

Core Lib JS Required

Please note that Web Skin was designed under the assumption that its consumer applications would be including the JavaScript assets in libs.js and libs_ie.js in their project(s). See the basic template section below for how these should be included in your HTML.

Basic template

Start with this basic HTML template, and adapt it to suit your needs.

<!DOCTYPE html>
<html>
    <head>
        <title>Wdesk Template Using Web Skin</title>
        <meta name="viewport" content="width=device-width, initial-scale=1.0">

        <!-- 
            NOTE: CONDITIONAL COMMENTS ARE NOT SUPPORTED IN MSIE 10 
            SERVER-SIDE MSIE DETECTION IS REQUIRED (not shown here)
        -->
        <!--[if !IE]> -->
        <link href="css/wdesk.min.css" rel="stylesheet" media="screen">
        <script src="js/jquery.min.js"></script>
        <script src="js/libs.min.js"></script>
        <!-- <![endif]-->
        <!--[if IE]>
        <link href="css/wdesk_ie.min.css" rel="stylesheet" media="screen">
        <script src="js/jquery.min.js"></script>
        <script src="js/libs_ie.min.js"></script>
        <![endif]-->
    </head>
    <body>
        <h1>Hello, WebFilers!</h1>

        <!-- Include all compiled plugins (below), or include individual files as needed -->
        <script src="js/web-skin.min.js"></script>
    </body>
</html>

Extending with Sass

Make building your app's SCSS easier by extending Web Skin's build-in SCSS variables, mixins, functions and more.

Once downloaded or installed via Bower, set up your app's Compass / SCSS accordingly. The following example assumes the following file structure

The first thing you need to do is make sure your Compass configuration file knows where to find the web-skin Sass files. Add the following line to your config.rb file.

additional_import_paths = ["bower_components/web-skin/sass"]

Once you've added this, you can import the web-skin Sass constants so they can be referenced throughout your app styles, and your app will update any time the values change in web-skin.

//---------------------------------------
//  constants
//---------------------------------------
@import "_constants/_support.scss";
@import "_constants/_wdesk-manifest.scss";

// Now you can write some responsive styles using the same settings 
// as the global packages uses so you're always in lock-step!
@media (min-width: $screenTablet) and (max-width: $screenTabletMax) {
    // mobile styles here
}

// Or make an element in your app be "Web Skin" Green
.my-element {
    background-color: $wf_green;
}

Additional notes

We've made many underlying changes in 0.2.x that are not immediately apparent. Base classes, key styles, and behaviors have been adjusted for flexibility and our mobile first approach.

Navs

  • You can no longer add "disabled" appearance to a nav link/button by using the parent <li class="disabled"> element. Instead, add the disabled class to an <a> element, or add the disabled attribute to a <button> element.

Forms

  • Text-based form controls no longer receive styling without adding the .form-control class.
  • Text-based form controls are now 100% wide, and cannot have grid classes applied directly to them. Wrap inputs inside <div class="col-*"></div> to control input widths.
  • .form-horizontal control groups no longer receive default widths. Use .form-group elems as grid rows, and wrap your inputs in grid spans to enforce a specific width as shown in this form-horizontal example.

Modals

  • Modal markup has changed significantly. The .modal-header, .modal-body, and .modal-footer sections now get wrapped in .modal-content and .modal-dialog for improved mobile styling and behavior.

Accordions

  • The accordion element classes shown in the Collapse Plugin JavaScript Docs of v0.1.x are no longer in-use. Instead, use the new Panels component to build accordions.

Javascript

  • JavaScript events are namespaced. For example, to handle the modal "show" event, use 'show.wdesk.modal'. For tabs "shown" use 'shown.wdesk.tab', etc.

For more information on upgrading, give us a shout or open an issue.

Browser support

Web Skin is built to work best in the latest desktop and mobile browsers, meaning older and less advanced browsers might receive a less stylized, though fully functional versions of certain components.

Supported browsers

Specifically, we support the latest versions of the following:

  • Safari +
  • Firefox +
  • MSIE +

Internet Explorer 7, 8 and 9

Internet Explorer 7, 8 and 9 are supported, however, please be aware that many CSS3 propertiesâe.g., rounded corners and shadowsâare not supported by IE7 or IE8. The placeholder attribute is also not supported in either of these versions.

In addition, Internet Explorer 7 and 8 requires the use of respond.js to enable media query support.

IE Compatibility modes

Web Skin is not supported in the old Internet Explorer compatibility modes. To be sure you're using the latest rendering mode for IE, consider including the appropriate <meta> tag in your pages:

<meta http-equiv="X-UA-Compatible" content="IE=edge">

See this StackOverflow question for more information.

Windows Phone 8 and Internet Explorer 10

Internet Explorer 10 doesn't differentiate device width from viewport width, and thus doesn't properly apply the media queries in Web Skin's CSS. To address this, you can optionally JavaScript to work alongside the CSS that is included with Web Skin until Microsoft issues a fix.

// Add this to work alongside Web Skin's CSS
if (navigator.userAgent.match(/IEMobile\/10\.0/)) {
    var msViewportStyle = document.createElement("style")
    msViewportStyle.appendChild(
        document.createTextNode(
            "@-ms-viewport{width:auto!important}"
        )
    )
    document.getElementsByTagName("head")[0].appendChild(msViewportStyle)
}
/* This comes with wdesk_ie.css */ 
@-ms-viewport { 
    width: device-width; 
}
@media screen and (max-width: 400px) {
    @-ms-viewport{
        width: 320px;
    }
}

For more information and usage guidelines, read Windows Phone 8 and Device-Width.

Modals and mobile devices

Overflow and scrolling

Support for overflow: hidden on the <body> element is quite limited in iOS and Android. To that end, when you scroll past the top or bottom of a modal in either of those devices' browsers, the <body> content will begin to scroll.

Virtual keyboards

Also, note that if you're using inputs in your modal â iOS has a rendering bug that doesn't update the position of fixed elements when the virtual keyboard is triggered. A few work arounds for this include transforming your elements to position: absolute or invoking a timer on focus to try to correct the positioning manually. This is not handled by Web Skin, so it is up to you to decide which solution is best for your application.

Browser zooming

Page zooming inevitably presents rendering artifacts in some components, both in Web Skin and the rest of the web. Depending on the issue, we may be able to fix it (search first and then open an issue if need be). However, we tend to ignore these as they often have no direct solution other than hacky workarounds.

Disabling responsiveness

Don't want your application to be scaled on different devices? With a little bit of work, you can disable the responsive features of Web Skin so that mobile users see your full desktop-version site.

Steps to disable responsive views

  1. Remove (or just don't add) the viewport <meta> mentioned in the CSS docs
  2. Remove the max-width on the .container for all grid tiers with max-width: none !important; and set a regular width. Be sure that this comes after the default Web Skin CSS. You can optionally avoid the !important with media queries or some selector-fu.
  3. If using navbars, undo all the navbar collapsing and expanding behavior.
  4. For grid layouts, make use of .col-xs-* classes in addition to or in place of the medium/large ones. Don't worry, the extra-small device grid scales up to all resolutions, so you're set there.

You'll still need respond.js for MSIE8 (since our media queries are still there and need to be picked up).

// Reset the container
.container {
    max-width: none !important;
    width: $screenDesktop;
}

.container .navbar-header,
.container .navbar-collapse {
    margin-right: 0;
    margin-left: 0;
}

// Always float the navbar header
.navbar-header { float: left; }

// Undo the collapsing navbar
.navbar-collapse {
    display: block !important;
    height: auto !important;
    padding-bottom: 0;
    overflow: visible !important;
}
.navbar-toggle { display: none; }
.navbar-brand { margin-left: -#{$gridGutterWidth / 2}; }

// Always apply the floated nav
.navbar .nav {
    float: left;
    margin: 0;
    > li { 
        float: left; 
        > .hitarea {
            padding: 0 nth($navbarPadding, 2);    
        }
    }
}

// Redeclare since we override the float above
.navbar .nav.navbar-right { float: right; }

Accessibility

Web Skin follows common web standards, and with minimal extra effort, can be used to create sites that are accessibile to those using AT.

Skip navigation

If your navigation contains many links and comes before the main content in the DOM, add a Skip to content link immediately after your opening <body> tag. (read why)

<body>
    <a href="#content" class="sr-only">Skip to content</a>
    <div class="container" id="content">
        The main page content.
    </div>
</body>

Nested headings

When nesting headings (<h1> - <h6>), your primary document header should be an <h1>. Subsequent headings should make logical use of <h2> - <h6> such that screen readers can construct a table of contents for your pages.

Learn more at HTML CodeSniffer and Penn State's AccessAbility.
